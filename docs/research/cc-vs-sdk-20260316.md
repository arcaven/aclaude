# Claude Agent SDK vs Direct Anthropic SDK

**Date:** 2026-03-16
**Status:** Active — informs forestage auth and data architecture

## Context

forestage needs to delegate auth to Claude Code (OAuth login) while retaining
raw access to the audit trail, context window data, and per-turn usage. This
document captures what each SDK provides and why we use both.

## Auth Landscape

Anthropic blocks OAuth tokens from third-party tools (enforced Jan 2026).
There is no public OAuth client registration. The only legitimate paths:

- **API key** (`ANTHROPIC_API_KEY`) — sanctioned for third-party tools
- **Claude Agent SDK** (`@anthropic-ai/claude-agent-sdk`) — spawns Claude Code
  as a subprocess, inherits whatever auth Claude Code has (OAuth, API key,
  Bedrock, Vertex)
- **Cloud providers** — Bedrock (AWS IAM), Vertex AI (Google ADC)

We use the agent SDK as the primary session runner so users on Pro/Max plans
can use their subscription credits via Claude Code's OAuth login.

## What the Agent SDK Provides

The `query()` function returns an `AsyncGenerator<SDKMessage>`.

### SDKSystemMessage (type "system", subtype "init")

```
session_id, model, cwd, tools[], mcp_servers[], permissionMode,
claude_code_version, apiKeySource, betas[], agents[], slash_commands[],
skills[], plugins[], uuid
```

Does NOT contain: context_window_size (only in ModelUsage at session end).

### SDKAssistantMessage (type "assistant")

Full `BetaMessage` from `@anthropic-ai/sdk`:
- `content: Array<TextBlock | ToolUseBlock>` — includes tool name, id, input
- `usage: { input_tokens, output_tokens, cache_read_input_tokens?, cache_creation_input_tokens? }`
- `stop_reason`, `model`, `id`

### SDKPartialAssistantMessage (type "stream_event")

Raw Anthropic streaming events: `message_start`, `content_block_start`,
`content_block_delta`, `content_block_stop`, `message_delta`, `message_stop`.

### SDKResultMessage (type "result")

Aggregated session stats:
- `total_cost_usd`, `duration_ms`, `duration_api_ms`, `num_turns`
- `usage: { input_tokens, output_tokens, cache_* }` (aggregate)
- `modelUsage: { [model]: ModelUsage }` (per-model breakdown)
- `result` (final text), `structured_output`, `permission_denials[]`

### ModelUsage (per-model in SDKResultMessage)

```
inputTokens, outputTokens, cacheReadInputTokens, cacheCreationInputTokens,
webSearchRequests, costUSD, contextWindow (total size in tokens)
```

### SDKToolProgressMessage (type "tool_progress")

Real-time: `tool_use_id`, `tool_name`, `elapsed_time_seconds`.

### Hook Callbacks (options.hooks)

Same events as Claude Code's hook system:
- PreToolUse, PostToolUse, PostToolUseFailure
- SessionStart, SessionEnd, Stop
- Notification, UserPromptSubmit
- SubagentStart, SubagentStop, PreCompact, PermissionRequest

Hook inputs include `tool_name`, `tool_input`, `tool_response` (PostToolUse),
`session_id`, and common fields. These run as JS callbacks, not shell commands.

## What the Direct SDK Provides

`new Anthropic()` with `messages.create()` / `messages.stream()`:

- **Per-message usage** — `finalMessage().usage` on every response
- **Full message control** — inject system messages, manipulate history, custom retry
- **Custom tools** — define your own tool schemas and handlers
- **Raw API access** — model listing, API key management, batch operations
- **Fine-grained streaming** — per-event control over the response stream

## What We Lose Without Each

### Without Agent SDK

- No OAuth delegation (must use API key)
- No Claude Code's tool execution (Read, Edit, Bash, Glob, Grep, etc.)
- No agent loop (tool use → result → continue cycle)
- Must implement tool execution ourselves

### Without Direct SDK

- No per-turn token usage (only aggregate at session end)
- No raw API calls (model listing, key management)
- No message-level control (can't inject mid-conversation)
- No custom tool definitions
- Less granular context tracking (must wire hooks for per-event data)

## How Pennyfarthing Gets This Data

Pennyfarthing uses Claude Code's **shell hook system** — commands registered in
`.claude/settings.local.json` that receive JSON on stdin at lifecycle events.

Key data flow for context tracking:

```
Claude Code → stdin JSON → hook script
{
  "context_window": {
    "current_usage": {
      "input_tokens": N,
      "cache_creation_input_tokens": N,
      "cache_read_input_tokens": N
    },
    "context_window_size": N
  },
  "session_id": "...",
  "model": "...",
  "tool_name": "...",
  "tool_input": {...}
}
```

Context percentage: `(input + cache_creation + cache_read) / context_window_size * 100`

This data arrives on **every hook event** (PreToolUse, PostToolUse, Stop, etc.),
giving real-time context tracking. The statusline reads it; the context breaker
circuit-breaks at 80%.

Pennyfarthing's audit trail lives in `.session/checkpoints.log`:
```
timestamp|label|data
2025-03-16T14:23:45Z|session_state|agent=pm;story=8-3;phase=planning;sha=abc123
```

## Our Architecture: Both SDKs

| Layer | Package | Use |
|---|---|---|
| Session runner | `@anthropic-ai/claude-agent-sdk` | Auth delegation, agent loop, tool execution, streaming |
| Raw API access | `@anthropic-ai/sdk` | Per-turn usage, model listing, direct messages, custom tools |
| Audit & monitoring | Agent SDK `options.hooks` | Context tracking, tool interception, session lifecycle |

The agent SDK depends on `@anthropic-ai/sdk` internally, so there is no
dependency conflict. Both can coexist.

### Hook Wiring (replacing pennyfarthing's shell hooks)

Instead of shell scripts reading stdin, we use JS callbacks in `options.hooks`:

```typescript
query({
  prompt: input,
  options: {
    hooks: {
      PreToolUse: [{ callback: onPreToolUse }],
      PostToolUse: [{ callback: onPostToolUse }],
      Stop: [{ callback: onStop }],
    }
  }
})
```

This gives us the same data pennyfarthing gets, but in-process with type safety.
